# Feature: Catalog Identity Consolidation

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Consolidate project identity so that **catalog IDs (m0lz.XX) are the primary name everywhere**: website project cards, about page, GitHub profile README, and external repo READMEs. Old project names (PICE, MCP-Guard, Case Pilot) are demoted from brand-level identifiers to descriptive context. The writing/research pages retain methodology names (e.g., "PICE Framework") as content, not product branding.

## User Story

As a visitor navigating between m0lz.dev and GitHub, I want every project identified consistently by its catalog ID, so that the naming feels unified rather than showing confusing dual names (e.g., "PICE m0lz.02").

### Patterns to Follow

**Catalog identity principle**: The catalog ID IS the project name. Old names become descriptive context, not brand identifiers.

- ✅ "m0lz.02 — Structured AI coding workflow orchestrator"
- ✅ "m0lz.02 implements the PICE methodology"
- ❌ "PICE m0lz.02"
- ❌ "PICE — Structured AI coding workflow orchestrator"

**CLI/npm package names are software identifiers, not brand names.** The npm package `mcp-guard` and CLI command `mcp-guard` are technical references that stay. They appear in installation commands, code examples, and config files — these are accurate documentation of the actual software.

Should the "What I'm building" section use catalog IDs as link text (e.g., `**[m0lz.04](...)**`) or keep human-readable names with catalog IDs in parentheses?
