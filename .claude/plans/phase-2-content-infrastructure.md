## Critical Design Decision: MDX Library

**`next-mdx-remote` was archived on April 9, 2026.** The PRD references it, but it is no longer maintained.

**Replacement: Direct `@mdx-js/mdx` compilation.**

Instead of any wrapper library, we use the core MDX compiler directly:

```typescript
import { evaluate } from '@mdx-js/mdx'
import * as runtime from 'react/jsx-runtime'

const { default: Content } = await evaluate(mdxSource, {
  ...runtime,
  remarkPlugins: [remarkGfm],
  rehypePlugins: [[rehypePrettyCode, { theme: { dark: 'min-dark', light: 'min-light' } }]],
})
```

**Why this works:**
- `evaluate()` runs at build time in the Server Component (Node.js environment)
- Output is static HTML ‚Äî no runtime MDX compilation
- Compatible with `output: 'export'`
- Zero dependency on archived or wrapper packages
- `gray-matter` handles frontmatter separately (strip before compilation)

**Syntax highlighting theme:** Dual themes (`min-dark` / `min-light`) via `rehype-pretty-code`. These are the most monochrome-friendly Shiki themes. The plugin outputs CSS custom properties that swap with `.dark` / `:root`.

### Task 3: Create MDX Compilation Pipeline

**File:** `lib/mdx.ts`
**Action:** CREATE
**Description:** Thin wrapper around `@mdx-js/mdx` `evaluate()`. Takes raw MDX content (frontmatter already stripped), compiles with remark-gfm and rehype-pretty-code, returns a React component.

```typescript
import { evaluate } from '@mdx-js/mdx'
import * as runtime from 'react/jsx-runtime'
import remarkGfm from 'remark-gfm'
import rehypePrettyCode from 'rehype-pretty-code'
import type { MDXContent } from 'mdx/types'
import { mdxComponents } from '@/components/mdx-components'

export async function compileMDX(source: string): Promise<MDXContent> {
  const { default: Content } = await evaluate(source, {
    ...runtime,
    remarkPlugins: [remarkGfm],
    rehypePlugins: [
      [rehypePrettyCode, {
        theme: { dark: 'min-dark', light: 'min-light' },
        keepBackground: false,
      }],
    ],
  })
  return Content
}
```

**Note:** `keepBackground: false` lets our CSS tokens control backgrounds. The dual theme setup outputs `[data-theme="dark"]` and `[data-theme="light"]` selectors on `<code>` elements. We'll need corresponding CSS in `globals.css` to wire these to our `.dark` class.

**Depends on:** Task 1
**Validate:** `npx tsc --noEmit`

### Task 4: Create Code Block Component

**File:** `components/code-block.tsx`
**Action:** CREATE
**Description:** Client Component (`'use client'`) that wraps `<pre>` elements from rehype-pretty-code. Features:
- **Copy button**: Top-right corner, uses `navigator.clipboard.writeText()`. Shows "Copied!" feedback for 2 seconds.
- **File name tab**: If `data-rehype-pretty-code-title` attribute is present, render as a tab above the code block.
- Styling: Use Tailwind + CSS variable tokens. Geist Mono font for code. Subtle border with `--border` token.

The component is registered as the `pre` override in `mdx-components.tsx`. It receives the `<pre>` element as children and wraps it with copy functionality.

```typescript
'use client'

import { useRef, useState } from 'react'

export function CodeBlock({ children, ...props }: React.HTMLAttributes<HTMLPreElement>) {
  const preRef = useRef<HTMLPreElement>(null)
  const [copied, setCopied] = useState(false)

  const copy = async () => {
    const text = preRef.current?.textContent ?? ''
    await navigator.clipboard.writeText(text)
    setCopied(true)
    setTimeout(() => setCopied(false), 2000)
  }

  return (
    <div className="relative group">
      <button onClick={copy} className="..." aria-label="Copy code">
        {copied ? 'Copied!' : 'Copy'}
      </button>
      <pre ref={preRef} {...props}>{children}</pre>
    </div>
  )
}
```

## Notes

### Design Decisions

- **`@mdx-js/mdx` over `next-mdx-remote`**: The original PRD specified `next-mdx-remote`, but it was [archived on April 9, 2026](https://github.com/hashicorp/next-mdx-remote/releases). Direct compilation via `@mdx-js/mdx` is simpler, has no wrapper dependency, and is future-proof.
- **Dual Shiki themes (`min-dark`/`min-light`)**: Chosen for maximum monochrome compatibility. `rehype-pretty-code` outputs CSS variables that swap based on `data-theme` attributes. We wire these to our `.dark` CSS class.
- **`gray-matter` for frontmatter**: Proven, lightweight, handles YAML frontmatter in MDX files. Strips frontmatter before passing to MDX compiler.
- **File-based content**: Posts live in `content/posts/{slug}/index.mdx`. The blog agent (m0lz.01) commits directly to this directory. No database, no CMS.

### Risks

- **`evaluate()` + Turbopack**: The `@mdx-js/mdx` `evaluate()` function uses `new Function()` internally. This works in Node.js at build time but may need verification with Turbopack's build pipeline. Fallback: use `compile()` + `run()` separately.
- **rehype-pretty-code dual theme CSS**: Wiring Shiki's `data-theme` attributes to our `.dark` class requires CSS selectors. If the plugin's output format changes, the CSS will break. Pin the version.
- **Static export with dynamic routes**: `generateStaticParams()` must enumerate all slugs at build time. If the filesystem read fails, the build fails. This is intentional ‚Äî broken content should never deploy.
