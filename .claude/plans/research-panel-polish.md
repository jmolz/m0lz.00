# Feature: Research Panel Polish — Scoped TOC + Scrollbar + Layout

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Refactor the slide-out `ResearchPanel` so it no longer looks like an embedded iframe: widen the panel, hide the native browser scrollbar, and add a panel-scoped "spotlight" table of contents (the same visual pattern as the floating TOC on the full research page, but scoped to the panel's scroll container instead of the viewport).

## User Story

As a reader on a writing post, I want to open the attached research panel, see a clean two-column layout with a live scroll-tracked table of contents, so that I can navigate long research documents without leaving the post I am reading.

## Success Criteria

- [ ] Panel has no native browser scrollbar — uses the existing `.toc-scroll` style (or a derived class) for any scrollable region
- [ ] Panel displays a sticky in-panel TOC on the right side showing H2 and H3 headings from the research content, with active-section highlighting that updates as the user scrolls the panel
- [ ] TOC anchor clicks scroll the panel's content area to the target heading (not the main document viewport)
- [ ] Panel width is increased so content + TOC sidebar render comfortably on desktop; graceful fallback on mobile (TOC hidden, single column)
- [ ] No regression: the writing-post preview list of sections (outside the panel) still displays and still opens the panel when clicked
- [ ] No regression: backdrop click, Escape key, "Open full page →" link, and close ✕ still work
- [ ] `npm run lint && npx tsc --noEmit && npm run test && npm run build` all pass

### Patterns to Follow

**Client component header:**

```tsx
'use client'

import { useState, useEffect, useRef, type ReactNode } from 'react'
```

**IntersectionObserver with cleanup (from `table-of-contents.tsx:14-51`):**

```tsx
const observerRef = useRef<IntersectionObserver | null>(null)

useEffect(() => {
  if (!isOpen) return
  const container = scrollContainerRef.current
  const article = container?.querySelector('article')
  if (!container || !article) return

  const headings = article.querySelectorAll('h2[id], h3[id]')
  if (headings.length === 0) return

  const tocItems: TocItem[] = Array.from(headings).map((el) => ({
    id: el.id,
    text: el.textContent || '',
    level: el.tagName === 'H2' ? 2 : 3,
  }))
  setItems(tocItems)
  setActiveId(tocItems[0].id)

  observerRef.current = new IntersectionObserver(
    (entries) => {
      const visible = entries.filter((e) => e.isIntersecting)
      if (visible.length > 0) setActiveId(visible[0].target.id)
    },
    {
      root: container,              // ← key difference vs. viewport TOC
      rootMargin: '-80px 0px -60% 0px',
      threshold: 0,
    },
  )

  headings.forEach((h) => observerRef.current?.observe(h))

  return () => observerRef.current?.disconnect()
}, [isOpen])
```

**Theme-aware colors (CSS custom properties, no `dark:` variant):**

## Notes

**Design decisions:**

- **Why `max-w-5xl` and not `w-[85vw]`?** The site uses discrete Tailwind breakpoints and fixed max-widths throughout (`max-w-3xl` on content pages, `max-w-2xl` on prose). Using `max-w-5xl` keeps the vocabulary consistent and gives 1024px of panel — enough for a ~672px content column plus a ~192px TOC column with breathing room.
- **Why inline the TOC logic instead of extracting a `PanelTableOfContents` component?** The TOC is tightly coupled to the panel's `scrollContainerRef` and `isOpen` state. Extracting it would require ref forwarding and a prop-passing dance for one caller. Three similar lines of code is better than a premature abstraction (per CLAUDE.md).
- **Why reuse `.toc-scroll` and not create a new `.panel-scroll`?** They want the same visual treatment. Creating a new class duplicates CSS for no reason.
- **Why not also render the floating `TableOfContents` inside the panel?** That component is hard-coded to use the viewport as the observer root. Dropping it into a fixed overlay would either do nothing (wrong root) or flicker (wrong rootMargin). Writing a panel-specific TOC is cheaper and more correct.
- **Why keep the outside preview list of section titles?** It advertises the research content on the writing post page without requiring the user to open the panel first. The user said "losing the spotlight TOC" not "losing the preview list" — the two serve different purposes.

**Trade-offs:**

- The panel now requires ≥1024px to show its TOC. Users on tablets and phones get a single-column layout, same as today. This is acceptable because the writing post page itself already hides its own TOC on those breakpoints (`hidden xl:block` on `table-of-contents.tsx`).
- The TOC inside the panel re-computes its item list every time the panel opens. This is cheap (one DOM query + one observer) and avoids stale state between opens.

**Potential follow-up work (out of scope for this plan):**

- Keyboard navigation inside the TOC (arrow keys to jump between entries)
- Animate the panel slide-in/out (currently instant via conditional render)
- Deep-link to a specific section: `/writing/pice-framework?research=seam-blindspot` opens the panel scrolled to that heading

**Known limitations:**

- If the research MDX has no `h2`/`h3` headings, the TOC will not render. This is the same behavior as `components/table-of-contents.tsx:53`.
- IntersectionObserver's `rootMargin: '-80px 0px -60% 0px'` may need tuning for the panel context — the `-80px` assumes a top offset for a nav bar that doesn't exist inside the panel. Revisit during manual testing.
