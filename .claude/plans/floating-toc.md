# Feature: Floating Table of Contents

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Add a floating/sticky table of contents sidebar that auto-generates from page headings (h2, h3). Visible on wide viewports while scrolling, highlights the active section via intersection observer, and collapses to hidden on narrow/mobile screens. Works on research pages, writing posts, and any content with headings.

## User Story

As a reader scrolling through a long research page (2,400+ lines), I want to see a persistent table of contents that tracks my position, so that I can jump between sections without scrolling to the top.

## Success Criteria

- [ ] Floating TOC visible on viewports >= 1280px, hidden below
- [ ] TOC auto-generates from h2/h3 headings in the rendered content
- [ ] Active section highlighted as user scrolls (IntersectionObserver)
- [ ] Clicking a TOC entry smooth-scrolls to that section
- [ ] Headings have stable `id` attributes for anchor linking (via rehype-slug)
- [ ] No layout shift when TOC appears/disappears
- [ ] Bundle stays under 100KB JS total
- [ ] No regressions in existing functionality
- [ ] Monochrome design: uses only --foreground, --muted, --border tokens

## Feature Metadata

## Implementation Tasks

Execute in order, top to bottom. Each task is atomic.

### Task 1: Add rehype-slug to MDX pipeline

**File:** `lib/mdx.ts`
**Action:** MODIFY
**Description:** Import `rehype-slug` and add it to the `rehypePlugins` array before `rehype-pretty-code`. This auto-generates `id` attributes on all headings (h1-h6) based on their text content. These IDs are what the floating TOC links to.
**Depends on:** none
**Validate:** `npm run build` — no errors; inspect built HTML for heading `id` attributes

### Task 2: Preserve heading IDs in MDX components

**File:** `components/mdx-components.tsx`
**Action:** MODIFY
**Description:** The current h2/h3 component overrides spread `{...props}` which should already pass through the `id` from rehype-slug. Verify this works. If not, destructure `id` explicitly and apply it. The key requirement: rendered `<h2 id="seam-blindspot">` etc. must survive the MDX component override.
**Depends on:** Task 1
**Validate:** `npm run build` + inspect HTML output for `id` attributes on headings

### Task 3: Add scroll-behavior CSS

**File:** `app/globals.css`
**Action:** MODIFY
**Description:** Add `html { scroll-behavior: smooth; }` and `scroll-padding-top: 5rem;` (to account for the sticky nav). This ensures anchor clicks produce smooth scrolling and don't hide content behind the nav bar.
**Depends on:** none
**Validate:** Manual — clicking an anchor link smooth-scrolls

### Task 4: Create TableOfContents client component

**File:** `components/table-of-contents.tsx`
**Action:** CREATE
**Description:** A `'use client'` component that:
1. On mount, queries the article container for all `h2[id]` and `h3[id]` elements
2. Builds a TOC list: `{ id, text, level }[]`
3. Sets up an IntersectionObserver on each heading to track which section is active
4. Renders a `<nav>` with:
   - Fixed position on the right side of the viewport
   - Only visible on `xl:` (>= 1280px) — use Tailwind `hidden xl:block`
   - A thin left border line (--border)
   - h2 items at normal indent, h3 items slightly indented
   - Active item in --foreground, others in --muted
   - Clicking an entry navigates to `#id`
5. Cleans up the observer on unmount

**Design details:**
- Position: `fixed right-[max(1rem,calc((100vw-48rem)/2-14rem))]` or similar to sit outside the content column on wide screens
- Width: ~200px max
- Font: text-xs, Geist Sans
- Top offset: ~6rem (below nav)
- Max height: `calc(100vh - 8rem)` with overflow-y-auto for very long TOCs
- No background — just text floating over the page background

**Depends on:** Task 1 (headings need IDs)
**Validate:** `npx tsc --noEmit` — no type errors; visual check on wide viewport

## Notes

### Design Decisions

- **Right-side positioning**: The content column is max-w-3xl (48rem = 768px). On a 1280px+ viewport there's ample space to the right. The TOC floats in that gutter.
- **No background/card**: The TOC is just text floating over the page background. Matches the site's minimalist design — no extra visual weight.
- **Auto-detection from DOM**: Rather than passing heading data from server to client (which would require serializing the MDX AST), the TOC reads headings from the DOM after mount. This is simpler and works with any content.
- **Hidden on mobile**: The TOC is a progressive enhancement. Mobile users still have the full reading experience; they just scroll.
- **rehype-slug over manual IDs**: Using rehype-slug ensures every heading across the entire site gets a stable ID. No manual anchor management needed.

### Risks

- **PICE research has 40+ headings**: The TOC might be very long. Mitigate with max-height + overflow-y-auto.
- **Bundle budget**: The IntersectionObserver + useState/useEffect is minimal JS. Should stay well under budget.
- **Heading text collisions**: rehype-slug handles duplicate headings by appending `-1`, `-2` etc.

### Open Questions

1. Should the TOC include h4 headings or just h2/h3? Starting with h2/h3 — can always extend.
2. Should the research page header's static CONTENTS be fully removed or kept as a mobile-only fallback? Plan says remove to avoid duplication.
