# Feature: Footer Distribution Links (NPM, Dev.to, Medium, Substack, Hacker News)

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Add 5 new outbound links to the site footer alongside the existing GitHub, LinkedIn, and RSS entries: NPM profile, Dev.to, Medium, Substack (subscribe), and Hacker News. Each is a plain anchor — no embedded forms, no third-party JS, no design-system exceptions. The footer renders on every page via the root layout, so adding links here surfaces the new distribution channels site-wide in one place.

## User Story

As a visitor reading m0lz.dev, I want to subscribe to new posts via Substack and discover Jacob's presence on other distribution platforms (NPM, Dev.to, Medium, Hacker News), so that I can follow his work on whichever platform I prefer.

### Critical (resolved before proceeding)

1. **Hidden assumption — Substack URL contains UTM parameters in the user's source.** The user pasted `https://m0lz.substack.com/?utm_campaign=pub&utm_medium=web`. UTM params on an outbound link from your own site are self-referential noise (you'd be attributing your own traffic as "web/pub") and ugly in the rendered HTML. — section: Task 2 — resolution: Stripped UTM params; the canonical Substack URL `https://m0lz.substack.com/` is used in Task 2. Documented in Notes.

2. **Weak success criterion — "Bundle size delta ≤ 0.5KB" has no validation command.** Without a measurement, this is aspirational. — section: Success Criteria + Contract — resolution: Replaced with the existing CLAUDE.md gate "Total JS bundle remains <100KB" which is the actual project constraint. The delta from 8 anchor tags is negligible by inspection (HTML-only, no JS); precise measurement is unnecessary at Tier 1.

3. **Verification-loop readiness — Task 4 (manual responsive check) blocks autonomous executor.** — section: Task 4 — resolution: Acknowledged. This is a UI feature where pixel-level wrapping cannot be verified by a headless test without snapshot tooling that doesn't exist in this repo. The executor stalls here intentionally and surfaces the dev URL to the user. Tier 1 plans permit this; alternative would be over-engineering a Playwright harness for a 5-link change.

### Consider (flagged, not acted on)

1. **Scope drift — link ordering ("GitHub, NPM, LinkedIn, Hacker News, Medium, Dev.to, Substack, RSS") was decided by the planner, not the user.** — section: Task 2. The user said "simple, on the bottom with everything else" but did not specify order. The chosen grouping (profiles → writing-distribution → feed) is defensible but reversible — flag for user confirmation during execution if it matters.

2. **Overcomplication — extracting links into `data/social.ts` was offered earlier and rejected by the user's "simple" answer.** Plan correctly keeps everything inline. Listed here only to confirm the rejection is honored.

3. **Hidden assumption — Hacker News profile pages don't render meaningful info to logged-out visitors.** `news.ycombinator.com/user?id=jmolz12` shows karma, account age, and submissions/comments lists only when clicked through. This is fine, but worth noting that the link is lower-value than the others. The user explicitly asked for it, so it stays.

4. **Reasoning divergence — alternative is `flex-wrap` with the link cluster, vs. stacking on mobile (`flex-col md:flex-row`).** Plan chose `flex-wrap` to keep the desktop single-row appearance and let wrapping happen naturally only when width forces it. Alternative would be more deterministic but adds a media-query breakpoint that the rest of the chrome doesn't use.

### Reasoning divergence (alternative paths not taken)

1. **Subscribe CTA at end of each post.** Higher-intent placement than footer. Rejected per user's "simple, on the bottom with everything else" answer. Worth revisiting after the next 2-3 posts ship if Substack conversion is low.

2. **Centralized `data/social.ts` for link metadata.** Would reduce repetition if links later appear in About page + post pages. Rejected for now as YAGNI — the only consumer is the footer, and a 33-line component is below the refactor threshold.

## Notes

### Design decisions

- **Substack is a plain link, not an embed.** The Substack official embed is an iframe with their styling, which directly violates the monochrome 4-token color system and the "no decorative elements" rule. A custom email form posting to Substack's undocumented endpoint is fragile and adds JS. The link-out keeps the footer pure and pushes form rendering to Substack where it's a solved problem.
- **UTM parameters stripped from the Substack URL.** Self-referential attribution on an outbound link from your own site is noise. The canonical `https://m0lz.substack.com/` is used.
- **Footer-only placement, no About page edit.** Per user: "the footer is on every page right?" — yes (rendered in `app/layout.tsx`), so the footer reaches every visitor without duplicating the link list anywhere else.
- **`flex-wrap` over breakpoint-based stacking.** Lets the link cluster gracefully wrap onto a second line on narrow viewports without imposing a hard breakpoint, matching the project's "design is symmetric" ethos.

### Trade-offs

- 8 footer links is near the upper limit of what feels uncluttered. If a 9th distribution channel is added later, consider extracting to `data/social.ts` and rendering a dedicated "Find me elsewhere" section on the About page.
- Hacker News profile page is lower-information than NPM/Dev.to/Medium — the user accepted this because the `jmolz12` handle can't be changed (their note).

### Potential follow-up work

- After 2-3 posts ship via Substack, measure conversion. If low, consider a small "Subscribe" CTA at end of each MDX post.
- Consider adding `data/social.ts` if/when a 9th link is added or if the About page wants to mirror the list.
- Mastodon, Threads, Bluesky links if/when those profiles exist.

### Known limitations

- No server-side click tracking — visitor analytics for outbound clicks rely on Vercel Analytics' default outbound-link telemetry (which is automatic on the existing setup).
