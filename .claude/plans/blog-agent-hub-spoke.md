# Feature: Blog Agent Hub-Spoke Content Architecture

> Generated by `/plan-feature` — read CLAUDE.md and referenced files before implementing.

## Overview

Build the blog agent (m0lz.01) as a CLI that publishes all content to m0lz.dev as the canonical hub, with project repos and cross-post platforms as spokes. The hub-spoke model ensures m0lz.dev owns all SEO authority, content is never duplicated across repos, and the blog agent has a single publish target.

## User Story

As a developer-author, I want to run `blog publish <slug>` and have the agent commit MDX to m0lz.00, set canonical URLs, cross-post to Medium/Dev.to, and update project READMEs with links back to m0lz.dev, so that all my content builds authority on one domain while reaching audiences on multiple platforms.

## Success Criteria

- [ ] Blog agent commits MDX posts to `m0lz.00/content/posts/{slug}/index.mdx`
- [ ] Every post frontmatter includes `project` field when linked to a catalog project
- [ ] Cross-posts set `canonical_url` / `canonicalUrl` pointing to `https://m0lz.dev/writing/{slug}`
- [ ] Project repo READMEs updated with links to m0lz.dev content (not local markdown)
- [ ] Research page on m0lz.dev correctly groups new posts under their catalog project
- [ ] No content duplication — project repos contain zero research/writing markdown files
- [ ] All validation commands pass with zero errors
- [ ] No regressions in existing functionality

### Patterns to Follow

**PostFrontmatter schema (the contract between agent and site):**

```yaml
---
title: "PICE: Structured AI Coding Workflows"
description: "Plan, Implement, Contract-Evaluate..."
date: "2026-04-10"
tags: ["ai", "developer-tools"]
canonical: "https://m0lz.dev/writing/pice-framework"
companion_repo: "https://github.com/jmolz/pice-framework"
project: "m0lz.02"
published: true
---
```

**Project linking (the hub-spoke glue):**

```
PostFrontmatter.project  ←→  ProjectData.catalogId
```

When `project: "m0lz.02"` appears in frontmatter, the Research page on m0lz.dev groups that post under the PICE project section with its branch mark, description, and repo link.

**MDX constraints:**
- No `<` characters in prose (MDX interprets as JSX) — use "under" instead
- No emojis (design constraint)
- Code blocks use standard fenced syntax
- Tables use standard markdown pipes

**Naming:**
- Post slugs: `kebab-case` matching the content/posts directory name
- Canonical URLs: `https://m0lz.dev/writing/{slug}`
- Project field values: catalog IDs exactly matching `data/projects.ts` (`m0lz.00`, `m0lz.02`, `m0lz.03`)

## Architecture: Hub-Spoke Content Model

### The Hub (m0lz.dev / m0lz.00)

```
m0lz.00/content/posts/
├── hey-im-jacob/index.mdx          # No project link
├── pice-framework/index.mdx        # project: "m0lz.02"
├── mcp-guard/index.mdx             # project: "m0lz.03"
├── case-pilot/index.mdx            # No project link (not in catalog)
├── investor-matchmaker/index.mdx   # No project link (not in catalog)
└── {future-post}/index.mdx         # Agent commits here
```

All content lives here. The site is the single source of truth. Every URL under `m0lz.dev/writing/*` is a canonical URL that other platforms point to.

### The Spokes

#### Spoke 1: Project Repos (link back to hub)

Project READMEs link to m0lz.dev for deep content instead of keeping markdown docs in their own repos.

**Before (wrong — content duplicated in repo):**
```
pice-framework/
├── README.md
├── docs/
│   ├── research/
│   │   ├── seam-blindspot.md       ← content lives in repo
│   │   └── convergence-analysis.md ← content lives in repo
│   └── providers/
└── src/
```

**After (right — repo links to hub):**
```
pice-framework/
├── README.md                       ← links to m0lz.dev/writing/...
├── docs/
│   └── providers/                  ← only API/usage docs stay in repo
└── src/
```

README sections like "Research" or "Deep Dives" become link lists:
```markdown
## Research

- [Structured AI Coding Workflows](https://m0lz.dev/writing/pice-framework)
- [Seam Blindspot Analysis](https://m0lz.dev/writing/seam-blindspot)
- [Convergence Analysis](https://m0lz.dev/writing/convergence-analysis)
```

#### Spoke 2: Cross-Post Platforms (canonical back to hub)

```
Dev.to article:
  canonical_url: "https://m0lz.dev/writing/pice-framework"

Medium story:
  canonicalUrl: "https://m0lz.dev/writing/pice-framework"
```

Both platforms display the content but search engines attribute authority to m0lz.dev.

#### Spoke 3: Social Media (link to hub)

LinkedIn posts, Twitter threads — all link to m0lz.dev/writing/{slug}.

### Data Flow

```
blog research <topic>
  → .blog-agent/research/{slug}.md (local only, never committed to site)

blog draft <slug>
  → .blog-agent/drafts/{slug}.mdx (local working copy)

blog publish <slug>
  1. Copy MDX to m0lz.00/content/posts/{slug}/index.mdx
  2. git commit + push → Vercel auto-deploys
  3. Wait for m0lz.dev/writing/{slug} to be live (poll Vercel API)
  4. Cross-post to Dev.to with canonical_url
  5. Cross-post to Medium with canonicalUrl (fallback to paste file)
  6. Update frontmatter with medium_url + devto_url
  7. If project: update project repo README with link to new post
  8. Generate social post drafts
```

## Notes

### Design Decisions

1. **Single repo publish target**: The agent always commits to m0lz.00. No content in spoke repos. This eliminates sync issues and content drift.
2. **Canonical URL is king**: Every cross-post, every README link, every social share points to m0lz.dev. Google sees one authoritative source.
3. **Project field = catalog glue**: The `project` frontmatter field uses catalog IDs (`m0lz.02`, `m0lz.03`) to link content to projects. This is the same field the Research page uses for grouping.
4. **Medium fallback is structural**: The agent generates a paste-ready markdown file when the Medium API fails. The pipeline never blocks on Medium.
5. **README updates are additive**: The agent appends links to README sections. It never rewrites README content. Idempotent.

### Learnings from Hub-Spoke Foundation Work

These patterns were established during the m0lz.00 build and must be preserved:

- MDX cannot contain bare `<` characters — they're interpreted as JSX tags. Use "under" instead of `<`.
- PostFrontmatter `project` field links to `ProjectData.catalogId` in `data/projects.ts`.
- Research page at `app/research/page.tsx` filters posts by `project` field and groups under catalog projects with branch marks.
- Post URLs are always `/writing/{slug}` regardless of whether they're "writing" or "research" — the Research page is a view, not a separate content type.
- Posts without a `project` field don't appear on the Research page — they're writing-only.

### Open Questions

1. Should the blog agent handle asset management (images, diagrams)? The current post structure supports `content/posts/{slug}/assets/` but no posts use it yet.
2. Should cross-posts strip companion_repo references since those are site-specific?
3. What's the tagging taxonomy? Currently freeform strings. Should the agent enforce a controlled vocabulary?
4. When PICE repos rename (pice-framework → m0lz.02), the agent needs to update README links and catalog references. Include migration tooling?
