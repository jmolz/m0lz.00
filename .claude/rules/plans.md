# Plans Directory Conventions

## Purpose

Plans in `.claude/plans/` are implementation plans generated by `/plan-feature`. They define what to build, how to build it, and the contract for evaluating success. Plans are the input to `/execute` and `/evaluate`.

## File Format

Plans follow the template at `.claude/templates/plan-template.md`. Every plan must include:

- **Overview** — what this implements and why
- **Context References** — must-read files, new files to create, patterns to follow
- **Implementation Tasks** — ordered, atomic tasks with file paths, actions, and validation commands
- **Testing Strategy** — unit, integration, and E2E steps
- **Validation Commands** — leveled from syntax/types through manual verification
- **Contract** — testable success criteria with tier, thresholds, and validation methods

## Naming

- Use kebab-case descriptive names: `add-user-auth.md`, `refactor-data-pipeline.md`
- The name should clearly identify the feature or change

## Contract Section

Every plan must end with a Contract section containing:

- **Tier** (1, 2, or 3) — determines evaluation depth
- **Success Criteria** as JSON with `name`, `threshold`, and `validation` for each criterion
- **Pass threshold** — minimum score (typically 7)

## Lifecycle

1. **Generated** by `/plan-feature` (or manually written)
2. **Negotiated** with the user — contract criteria are agreed upon before implementation
3. **Consumed** by `/execute` — tasks are implemented in order
4. **Graded** by `/evaluate` — contract criteria are scored by an adversarial evaluator
5. **Archived** — kept as project record after implementation

## Version Control

- Plans are generated artifacts — commit only when they serve as project record
- Do not modify a plan after `/execute` has started — create a new plan if scope changes
- Reference plans by path in commit messages when relevant
